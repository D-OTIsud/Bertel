# Object Data Dictionary — Bertel V3

**Type:** Reference document  
**Status:** Canonical for V1  
**Sources:** `schema_unified.sql`, `api_views_functions.sql`, `rls_policies.sql`, `seeds_data.sql`, `ui_whitelabel_branding.sql`, `migration_sustainability_v5.sql`, `bertel-object-workspace-canonical-map.md`, `domain.ts`, `bertel-tourism-ui/src/types/`, `bertel-tourism-ui/src/services/`  
**Purpose:** Single source of truth describing every data capacity an Object can hold, its supporting infrastructure (users, actors, security, performance), and how it maps to business logic, API functions, and UI panels.

**Coverage note:** This dictionary documents object-facing data plus the governance infrastructure that changes visibility, permissions, publication behavior, synchronization, and API payload shape. Repetitive physical implementations such as `ref_code_*` partitions and default partitions are documented through their canonical parent model unless a child table introduces distinct business semantics. For narrow junction tables, the focus is on business-significant columns and invariants rather than repeating generic audit timestamps on every row.

## 1. The Concept of an "Object"

An **Object** (`object` table) is the universal root entity in Bertel. It represents any tourism entity that the system can describe, publish, moderate, and relate to other entities. This includes physical establishments (hotels, restaurants, campsites), experiential products (guided activities, events), geographic references (itineraries, points of interest), and institutional structures (tourism offices).

Every Object has a `object_type` that determines which conditional data modules apply to it. The type is a hard classification—not a tag—and drives API filtering, conditional UI panels, and business rule enforcement.

**Object types (enum `object_type`):**

| Code | Meaning |
|------|---------|
| `HOT` | Hotel / classified accommodation |
| `HPA` | Para-hotel accommodation (apartments, residences) |
| `HLO` | Holiday letting (gîtes, vacation rentals) |
| `CAMP` | Campsite / outdoor accommodation |
| `RVA` | Résidence de vacances |
| `RES` | Restaurant / food service |
| `ITI` | Itinerary / hiking trail |
| `FMA` | Fête / manifestation / event |
| `ACT` | Supervised commercial activity (guided tours, sports sessions) |
| `LOI` | Leisure / attraction |
| `PCU` | Cultural heritage point |
| `PNA` | Natural site / outdoor area |
| `VIL` | Village / locality |
| `COM` | Commerce / retail |
| `PSV` | Service provider |
| `ASC` | Association |
| `ORG` | Organizational entity (tourism office, publisher) |

The `secondary_types` column (transitional, added Lot 1 2026-03-20) allows multi-belonging for pilot objects. It is non-canonical and opt-in only—no API filters it automatically. Replacement condition: if multi-belonging exceeds 50 objects with confirmed UI need, migrate to a junction table.

### Panel A1 — General Identity (`object` table)

**Concept:** The root identity record of any Object. Every other table cascades from this.

**Business Purpose:** Establishes the minimal facts needed to create, route, and control an entity: its type, name, operational timezone, publication status, and membership-driven visibility. These fields govern everything from editorial workflow to search indexing.

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `id` | `TEXT` | Yes | Structured identifier in format `{TYPE}{REGION}{SEQ}` (e.g., `HOTREU000V5014ZU`). Auto-generated by trigger on insert. |
| `object_type` | `object_type` (enum) | Yes | The hard type of this entity. Drives conditional modules, API filters, and UI panels. |
| `secondary_types` | `object_type[]` | No | Transitional: additional types for multi-belonging objects (Lot 1 pilot). Non-canonical; not filtered by any API. |
| `name` | `TEXT` | Yes | The canonical display name of the entity. Used in search, listings, and headers. |
| `name_i18n` | `JSONB` | No | Translated versions of the name: `{"fr": "...", "en": "...", "es": "..."}`. |
| `name_normalized` | `TEXT` (generated) | — | Unaccented lowercase name for case-insensitive search. Auto-maintained. |
| `name_search_vector` | `tsvector` (generated) | — | French full-text search vector over the name. Auto-maintained. |
| `business_timezone` | `TEXT` | Yes | IANA timezone for this entity (default: `Indian/Reunion`). Used when evaluating "open now" logic. Validated against `pg_timezone_names`. |
| `region_code` | `TEXT` | No | 3-character alphanumeric region identifier (e.g., `REU`). Used in ID generation. |
| `status` | `object_status` (enum) | Yes | Editorial lifecycle: `draft`, `published`, `archived`, `hidden`. Default: `draft`. |
| `updated_at_source` | `TIMESTAMPTZ` | No | Freshness timestamp coming from the upstream source system. Kept separate from business-edit timestamp `updated_at`. Incremental resource endpoints can paginate on this field when `p_use_source = true`. |
| `published_at` | `TIMESTAMPTZ` | No | Timestamp of first publication. Auto-set by trigger when `status` transitions to `published`. |
| `is_editing` | `BOOLEAN` | Yes | True when at least one `pending_change` with status `pending` exists for this object. Managed by database triggers. UI should show a "review in progress" indicator. |
| `commercial_visibility` | `TEXT` | Yes | Membership-driven visibility: `active`, `lapsed`, `suspended`. Controlled by `object_membership` triggers. Determines whether the object appears in commercial channels. |
| `cached_min_price` | `NUMERIC(10,2)` | No | Denormalized lowest active price from `object_price`. Updated by triggers. Used in listing cards to avoid expensive aggregations. |
| `cached_main_image_url` | `TEXT` | No | URL of the primary media item. Updated by triggers. Used in listing cards. |
| `cached_rating` | `NUMERIC(3,2)` | No | Aggregate review rating (0–5). Updated by triggers when `object_review` changes. |
| `cached_review_count` | `INTEGER` | No | Total count of published reviews. Updated by triggers. |
| `cached_is_open_now` | `BOOLEAN` | No | Whether the entity is currently open, evaluated against `opening_period`/`opening_schedule`. Updated by scheduled job. |
| `cached_amenity_codes` | `TEXT[]` | No | Denormalized array of amenity codes from `object_amenity`. Enables fast array-containment filter queries. |
| `cached_payment_codes` | `TEXT[]` | No | Denormalized array of accepted payment method codes. |
| `cached_environment_tags` | `TEXT[]` | No | Denormalized array of environment tag codes. |
| `cached_language_codes` | `TEXT[]` | No | Denormalized array of spoken language codes. |
| `cached_classification_codes` | `TEXT[]` | No | Denormalized array of composite distinction tokens encoded as `scheme_code:value_code`. Used by explorer filters and indexed with GIN. |
| `cached_taxonomy_codes` | `TEXT[]` | No | Denormalized array of composite taxonomy tokens encoded as `domain:code`, populated from assignable ancestor paths. Used by hierarchical explorer filters and indexed with GIN. |
| `current_version` | `INTEGER` | Yes | Monotonic version counter. Incremented on each versioned change. Avoids a `MAX()` scan on `object_version`. |
| `extra` | `JSONB` | No | Escape hatch for non-normalized metadata. Use sparingly; prefer adding a dedicated column. |
| `created_at` | `TIMESTAMPTZ` | Yes | Record creation timestamp. |
| `updated_at` | `TIMESTAMPTZ` | Yes | Business change timestamp. Updated only when non-cache fields change (see trigger `update_object_updated_at_business`). |
| `created_by` | `UUID` (FK → `auth.users`) | No | User who created the record. |
| `updated_by` | `UUID` (FK → `auth.users`) | No | User who last modified the record. |

### Panel A2 — Structuring Taxonomy (`object_taxonomy`, `object_classification`)

**Concept:** Two parallel classification systems:
- **Taxonomy** (`object_taxonomy` / `ref_code_taxonomy_closure`): Hierarchical classification for discovery filtering (e.g., accommodation sub-types, activity categories). Uses the `ref_code` partitioned table with closure table for ancestor queries.
- **Classification / Labels** (`object_classification` / `ref_classification_scheme` / `ref_classification_value`): Official quality marks, ratings, and certifications (e.g., star ratings, green labels, accessibility certifications).

**Business Purpose:** Taxonomy drives the explorer filter buckets and search relevance. Classifications surface as public badges and feed into visibility rules.

**`object_taxonomy` table:**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `id` | `UUID` | Yes | Primary key. |
| `object_id` | `TEXT` (FK) | Yes | The object being classified. |
| `domain` | `TEXT` | Yes | The taxonomy domain registered in `ref_code_domain_registry` (e.g., `hot_type`, `act_category`, `retail_category`). |
| `ref_code_id` | `UUID` (FK → `ref_code`) | Yes | The assigned taxonomy node within that domain. |
| `source` | `TEXT` | No | Origin of the assignment (manual, import, synchronization, etc.). |
| `note` | `TEXT` | No | Internal note explaining the assignment or exception. |
| `created_at` / `updated_at` | `TIMESTAMPTZ` | Yes | Audit timestamps. |

> **Invariant:** One taxonomy assignment per `(object_id, domain)`; the unique constraint prevents multiple concurrent assignments within the same domain.
>
> **API contract note:** Resource payloads expose `taxonomy` as a first-class block separate from `classifications`. Assignments are keyed by `(domain, ref_code_id)` and expanded through `ref_code_taxonomy_closure` to build full public paths for filtering and display.

**`object_classification` table:**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `object_id` | `TEXT` (FK) | Yes | The classified object. |
| `scheme_id` | `UUID` (FK → `ref_classification_scheme`) | Yes | The classification scheme (e.g., "Hotel Stars", "Green Key", "Tourisme & Handicap"). |
| `value_id` | `UUID` (FK → `ref_classification_value`) | Yes | The specific value within the scheme (e.g., "3 stars", "label granted"). |
| `subvalue_ids` | `UUID[]` | No | Optional sub-values for multi-dimensional schemes (e.g., disability categories for Tourisme & Handicap). |
| `document_id` | `UUID` (FK → `ref_document`) | No | Supporting document or certificate. |
| `requested_at` | `DATE` | No | When certification was requested. |
| `awarded_at` | `DATE` | No | When certification was officially granted. |
| `valid_until` | `DATE` | No | Expiry date of the certification. |
| `status` | `TEXT` | No | Lifecycle: `requested`, `granted`, `suspended`, `expired`. |
| `source` | `TEXT` | No | Origin of the data (import source, manual entry, etc.). |
| `note` | `TEXT` | No | Internal note about this classification. |

### Panel A3 — Publication & Moderation (`pending_change`, `object_version`)

**Concept:** A content moderation queue and version history. Changes proposed by external users (or internal agents with limited rights) are staged as `pending_change` records before being applied. Every significant change is versioned in `object_version`.

**Business Purpose:** Allows an Office de Tourisme to review and approve changes before they affect published data. Essential for data quality governance.

**`pending_change` table:**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `object_id` | `TEXT` (FK) | No | The object this change targets (may be null for non-object changes). |
| `target_table` | `TEXT` | Yes | Which table is being modified (e.g., `object_description`, `object_location`). |
| `target_pk` | `TEXT` | No | Primary key of the row being changed. |
| `action` | `TEXT` | Yes | Type of change: `insert`, `update`, or `delete`. |
| `payload` | `JSONB` | Yes | The proposed new values. |
| `submitted_by` | `UUID` (FK) | No | User who submitted the change. |
| `submitted_at` | `TIMESTAMPTZ` | Yes | Submission timestamp. |
| `status` | `TEXT` | Yes | Workflow state: `pending`, `approved`, `rejected`, `applied`. |
| `reviewed_by` | `UUID` (FK) | No | Moderator who reviewed the change. |
| `reviewed_at` | `TIMESTAMPTZ` | No | Review timestamp. |
| `review_note` | `TEXT` | No | Moderator's comment (reason for rejection, etc.). |
| `applied_at` | `TIMESTAMPTZ` | No | When the change was actually written to the target table. |
| `metadata` | `JSONB` | No | Additional context (client version, device, etc.). |

**`object_version` table (partitioned by `created_at`, monthly RANGE):**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `id` | `UUID` | Yes | Primary key (composite with `created_at`). |
| `object_id` | `TEXT` (FK) | Yes | The object being versioned. |
| `version_number` | `INTEGER` | Yes | Monotonic version counter (matches `object.current_version` at snapshot time). |
| `data` | `JSONB` | Yes | Full snapshot of the object state at this version. |
| `created_at` | `TIMESTAMPTZ` | Yes | When this version was created. Part of the composite PK and the partition key. |
| `created_by` | `UUID` (FK → `auth.users`) | No | User who triggered the version. |
| `change_reason` | `TEXT` | No | Human-readable reason for the change. |
| `change_type` | `TEXT` | Yes | `insert`, `update`, or `delete`. Default: `update`. |

> **Partitioning:** Monthly range partitions on `created_at`. Partitions are auto-created by `create_object_version_monthly_partition()` and a default partition catches inserts when the monthly partition is missing. Unique constraint on `(object_id, version_number, created_at)`.

### Panel A4 — Synchronization & External IDs (`object_external_id`, `object_origin`)

**Concept:** Tracks where an object came from and which external systems know it by what identifier.

**Business Purpose:** Enables bi-directional sync with source systems (e.g., SIT feeds, booking platforms). Prevents duplicate imports and enables re-sync without data loss.

**`object_external_id` table:**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `object_id` | `TEXT` (FK) | Yes | The Bertel object. |
| `organization_object_id` | `TEXT` (FK) | Yes | The ORG that owns this external reference (the SIT publisher). |
| `source_system` | `TEXT` | Yes | Identifier of the external system (default: `default`). |
| `external_id` | `TEXT` | Yes | The ID in the external system. |
| `last_synced_at` | `TIMESTAMPTZ` | No | Timestamp of last successful sync with the external system. |

**`object_origin` table (1:1):**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `object_id` | `TEXT` (PK/FK) | Yes | The object. |
| `source_system` | `TEXT` | No | Name of the originating system. |
| `source_object_id` | `TEXT` | No | ID in the source system at time of first import. |
| `import_batch_id` | `UUID` | No | Batch identifier for the import job that created this object. |
| `first_imported_at` | `TIMESTAMPTZ` | No | Timestamp of initial ingestion. |

### Panel A5 — Print Publications (`publication`, `publication_object`)

**Concept:** Manages the lifecycle of print guides, brochures, and annual directories. Tracks which objects are selected for each edition and where they are in the print production workflow.

**Business Purpose:** Tourism offices publish annual print guides. Each edition selects a set of objects, sends proofs to providers for validation (BAT — Bon à Tirer), and tracks page assignments. The proof-send action auto-creates a CRM interaction for provider follow-up.

**`publication` table:**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `id` | `UUID` | Yes | Primary key. |
| `code` | `TEXT` | Yes | Unique code, normalized (`immutable_unaccent(lower(code))`). |
| `name` | `TEXT` | Yes | Publication title (e.g., "Guide Touristique 2026"). |
| `target_audience` | `TEXT` | No | Intended audience description. |
| `year` | `INTEGER` | No | Publication year (2000–2200). |
| `status` | `TEXT` | Yes | Lifecycle: `planning`, `proofing`, `at_press`, `published`. Default: `planning`. |
| `metadata` | `JSONB` | No | Additional production metadata. |
| `created_by` / `updated_by` | `UUID` (FK) | No | Audit trail. |

**`publication_object` table (junction):**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `publication_id` | `UUID` (FK) | Yes | PK part 1. |
| `object_id` | `TEXT` (FK) | Yes | PK part 2. |
| `custom_print_text` | `TEXT` | No | Override description for this print edition. |
| `workflow_status` | `TEXT` | Yes | `selected`, `proof_sent`, `changes_requested`, `validated_bat`, `rejected`. Default: `selected`. |
| `page_number` | `INTEGER` | No | Assigned page in the publication. Must be > 0. |
| `proof_sent_at` | `TIMESTAMPTZ` | No | Auto-set by trigger when `workflow_status` transitions to `proof_sent`. |
| `validated_bat_at` | `TIMESTAMPTZ` | No | Auto-set by trigger when `workflow_status` transitions to `validated_bat`. |
| `metadata` | `JSONB` | No | Per-object production notes. |

> **Trigger:** `trg_publication_object_workflow_timestamps` auto-sets `proof_sent_at` and `validated_bat_at`. A second trigger `api.log_publication_proof_interaction` auto-creates a CRM interaction record when a proof is sent.

### Panel B1 — Location (`object_location`, `object_place`, `object_zone`)

**Concept:** A unified location model supporting both a single main GPS point and multiple named sub-locations ("places") within a complex site.

**Business Purpose:** Enables map rendering, geo-search (proximity, bounding box), address display, and commune-level statistics. Sub-places allow a hotel to have separate GPS points for its spa, pool, and restaurant.

**`object_place` table (sub-locations):**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `object_id` | `TEXT` (FK) | Yes | Parent object. |
| `label` | `TEXT` | No | Human-readable name for this place (e.g., "Main entrance", "Pool area"). |
| `slug` | `TEXT` | No | URL-safe identifier. |
| `is_primary` | `BOOLEAN` | No | True for the canonical sub-location. At most one primary per object (enforced by partial unique index). |
| `position` | `INTEGER` | No | Display order. |
| `effective_from` / `effective_to` | `DATE` | No | Date range during which this place is active (e.g., seasonal annex). |
| `extra` | `JSONB` | No | Extension field. |

**`object_location` table (unified address + GPS):**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `object_id` | `TEXT` (FK) | No* | Links to the parent object (XOR with `place_id`). |
| `place_id` | `UUID` (FK) | No* | Links to a sub-place (XOR with `object_id`). |
| `address1` | `TEXT` | No | Street address line 1. |
| `address1_suite` | `TEXT` | No | Address line 1 complement. |
| `address2` | `TEXT` | No | Street address line 2 (building, floor, etc.). |
| `address3` | `TEXT` | No | Additional address information. |
| `postcode` | `VARCHAR(10)` | No | Postal code. |
| `city` | `TEXT` | No | City name. |
| `code_insee` | `VARCHAR(5)` | No | French INSEE municipality code. Used for statistical aggregations. |
| `lieu_dit` | `TEXT` | No | Named locality / hamlet. |
| `direction` | `TEXT` | No | Directions or access notes (free text). |
| `zone_touristique` | `TEXT` | No | Transitional (Lot 1): free-text tourist zone. Will be replaced by FK to `ref_zone_touristique` when normalized. |
| `latitude` | `DECIMAL(10,8)` | No | WGS84 latitude. Must be paired with longitude. |
| `longitude` | `DECIMAL(11,8)` | No | WGS84 longitude. Must be paired with latitude. |
| `altitude_m` | `INTEGER` | No | Altitude in meters. Relevant for trails, mountain sites. |
| `is_main_location` | `BOOLEAN` | No | True for the canonical location record. At most one main per object/place (enforced by partial unique index). |
| `geog2` | `GEOGRAPHY(POINT,4326)` (generated) | — | PostGIS geography point auto-computed from lat/lon. Powers proximity queries. |
| `city_search_vector` | `tsvector` (generated) | — | Full-text search vector over city name. |

**`object_zone` table:**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `object_id` | `TEXT` (FK) | Yes | Parent object. |
| `insee_commune` | `VARCHAR(5)` | Yes | INSEE code of a commune this object belongs to. An object can span multiple communes. |
| `position` | `INTEGER` | No | Display order. |

### Panel B4 — Public Contacts (`contact_channel`)

**Concept:** Contact channels directly attached to the Object itself (as opposed to contacts attached to an Actor). These are the published contact details of the establishment.

**Business Purpose:** Populates the "Contact" section of public listings: phone numbers, email addresses, website URLs, social media profiles.

**`contact_channel` table:**

| Field | Type | Required | Purpose |

## 4. Relational Complexities

### 4.1 Object-to-ORG (Publisher) Link

```
object  ──(object_org_link)──  object[ORG]
         role: publisher
```

An Object belongs to a single primary ORG publisher (the entity that manages its data, typically an Office de Tourisme). This is enforced by a partial unique index on `object_org_link(object_id) WHERE is_primary`. Multiple secondary ORG links are allowed for partnerships.

### 4.2 Object-to-ACTOR (Operator) Link

```
object  ──(actor_object_role)──  actor
         role: operator / guide / manager / contact
```

An ACTOR is a person or commercial operator—never an institutional ORG. The canonical ACT pattern explicitly requires an ACTOR as operator. An Actor has its own contact channels (`actor_channel`) and RGPD consent records (`actor_consent`).

### 4.3 Object-to-Object Relations

```
object (source) ──(object_relation)──  object (target)
                  relation_type: based_at_site
                               | uses_itinerary
                               | child_of
                               | ...
```

Typed semantic relations between objects. Used for: ACT → PNA site, ACT → ITI trail, FMA → LOI venue, etc.

### 4.4 Location Scope (Object vs. Place)

```
object_location.object_id   → main GPS/address of the object itself
object_location.place_id    → GPS/address of a named sub-place
```

The `chk_object_or_place` constraint ensures exactly one of these is set. A restaurant's kitchen, dining room, and terrace can each have their own `object_place` record with its own `object_location` and `object_place_description`.

### 4.5 Description Scope (Object vs. ORG)

```
object_description.org_object_id IS NULL   → canonical description for all contexts
object_description.org_object_id IS NOT NULL → org-specific override (one per org)
```

Allows an ORG to maintain a customized description of an object for their specific publication, without overwriting the canonical version.

### 4.6 Membership → Commercial Visibility Cascade

```
object_membership.status (paid/invoiced) → object.commercial_visibility = 'active'
object_membership.status (lapsed)         → object.commercial_visibility = 'lapsed'
```

This cascade is implemented as a database trigger (`trg_membership_status_transition`). It applies to both object-level memberships and ORG-wide memberships (which propagate to all objects linked to the ORG via `object_org_link`).

### 4.7 Pending Changes → `is_editing` Flag

```
pending_change (status=pending, object_id=X)  → object[X].is_editing = TRUE
pending_change (no more pending for object X)  → object[X].is_editing = FALSE
```

Managed by triggers on `pending_change` INSERT/UPDATE/DELETE. The `is_editing` flag enables the UI to display a moderation-in-progress indicator on the object card.

| `value_text` | `TEXT` | No* | Text translation payload. |

### 5.9 — Access Scope, Role Assignment & Explicit Permissions (`user_org_business_role`, `user_org_admin_role`, `org_config`, `ref_permission`, `org_permission`, `user_permission`)

**Concept:** Access control is intentionally split into orthogonal layers:
- **Membership scope:** which ORG a user belongs to
- **Role labels:** business and admin roles attached to that membership
- **Explicit permissions:** grantable actions that do not automatically derive from role names

**Business Purpose:** Prevents role-title inflation from becoming a hidden permission system. A user may be an "editor" or local admin and still need an explicit permission such as `create_object` or `publish_object`.

**`user_org_business_role` / `user_org_admin_role` tables:** Assignment tables linking a membership to one active business role and one active admin role at most. Each row records `membership_id`, `role_id`, `is_active`, `assigned_by`, and `assigned_at`.

**`org_config` table (one row per ORG):**

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `org_object_id` | `TEXT` (FK → `object`) | Yes | The ORG this config belongs to. Must point to `object_type = 'ORG'`. |
| `access_scope` | `TEXT` | Yes | Scope mode: `own_objects_only` or `all_published`. Controls whether members can read only their own ORG perimeter or all published external objects as well. |
| `created_at` / `updated_at` | `TIMESTAMPTZ` | Yes | Audit timestamps. |

**`ref_permission` table:** Permission catalog. Key columns are `code`, `name`, `category`, `description`, and `is_active`. Categories are check-constrained (`content`, `crm`, `team`, `media`) instead of ENUM-based to keep migration flexibility.

**`org_permission` table:** ORG-wide permission grants inherited by all active members of that ORG. Unique on `(org_object_id, permission_id)` and soft-revocable via `is_active`.

**`user_permission` table:** Direct user-level additive grants. Unique on `(user_id, permission_id)` and evaluated ahead of ORG grants by `api.user_has_permission()`.

> **Key invariants:** Admin rank does not bypass explicit permissions. `org_config.access_scope = 'all_published'` widens read scope only; it does not create write rights on external objects.
>
> **Operational surface:** These tables are administered via dedicated SECURITY DEFINER RPCs such as `api.rpc_upsert_membership()`, `api.rpc_deactivate_membership()`, `api.rpc_set_business_role()`, `api.rpc_set_admin_role()`, `api.rpc_revoke_admin_role()`, `api.rpc_grant_org_permission()`, `api.rpc_revoke_org_permission()`, `api.rpc_grant_user_permission()`, and `api.rpc_revoke_user_permission()`.

## 8. Security Model (Row-Level Security)

All tables have RLS enabled. Access is governed by a three-tier model with helper functions.

---

### 8.1 — Access Tiers

| Tier | Who | What they see |
|------|-----|---------------|
| **Public** | Anonymous / unauthenticated | Only rows with `status = 'published'`, `is_published = TRUE`, `visibility = 'public'`, or fully open tables (locations, opening hours, classifications). |
| **Extended** | Authenticated users with org/actor link | All public data + draft/archived objects within their own org scope, plus optional cross-ORG published visibility when their active ORG has `org_config.access_scope = 'all_published'`. Evaluated by `api.can_read_extended(p_object_id)`. |
| **Admin** | `service_role`, `admin`, platform owner/super_admin | Full unrestricted access to all rows. |

### 8.3 — Policy Patterns by Table Category

| Table category | Read policy | Write policy |
|----------------|------------|--------------|
| **Object core** (`object`) | Published = public; extended via `can_read_extended`; admin = all | Owner (`created_by`) or admin |
| **Object child tables** (location, classification, capacity, amenity, language, payment, environment, zone, opening_*) | Fully public (`USING (true)`) | Owner via `is_object_owner` |
| **Visibility-gated** (description, media, contact_channel) | Public where `visibility='public'` / `is_published=true`; extended via `can_read_extended` | Owner via `is_object_owner` |
| **Private notes** (`object_private_description`) | Extended: same-org only | Insert: same-org + self as author; Update/Delete: hierarchical admin rank |
| **Legal records** | Extended via `can_read_extended` | Owner via `is_object_owner` |
| **Admin-curated overlays** (`object_place_description`, `object_origin`, `object_relation`) | Public read or extended read depending on table | Admin write only |
| **External IDs** (`object_external_id`) | Admin/service_role only | Admin/service_role only |
| **Reference tables** (`ref_*`) | Public read (`USING (true)`) | Admin write only |
| **User profile** | Self (`id = auth.uid()`) or platform owner | Self or platform owner |
| **Org scope / permission governance** (`user_org_membership`, `user_org_business_role`, `user_org_admin_role`, `org_config`, `org_permission`, `user_permission`) | Member or superuser scoped | Superuser/admin-managed depending on table |
| **CRM** (`crm_interaction`, `crm_task`) | Admin only | Admin only |
| **Audit** (`audit.audit_log`) | Admin only | Insert by `service_role`/`postgres` (triggers) |
| **Publications** | Admin only | Admin only |
| **Promotions** (`promotion`, `promotion_object`) | Public read; usage journal admin-only | Promotion definition admin write |
| **Reviews** | Published reviews only | Admin only |

> **Dynamic RLS on partitions:** A `DO` block iterates over all `ref_code` partitions and applies the standard public-read / admin-write policy pair to each. Same pattern for `object_version` and `audit.audit_log` partitions.

## 9. Performance Architecture

---

### 9.1 — Materialized Views

| View | Schema | Refresh | Purpose |
|------|--------|---------|---------|
| `internal.mv_ref_data_json` | `internal` | Daily or on ref data change | Pre-joined reference data (amenities with families, languages, media types, contact kinds) as JSONB. Avoids repeated joins in API functions. Indexed on `(ref_type, id)` and `(ref_type, code)`. |

| **GiST on geography** | `object_location.geog2` | Proximity queries (`ST_DWithin`), bounding-box filters. |
| **GIN on arrays** | `object.cached_amenity_codes`, `cached_payment_codes`, `cached_environment_tags`, `cached_language_codes`, `cached_classification_codes`, `cached_taxonomy_codes` | Array-containment filters (`@>`). |
| **Keyset pagination** | `object(object_type, status, updated_at DESC, id)` | Cursor-based pagination for list endpoints. |
| **Partial unique** | `object_org_link(object_id) WHERE is_primary`, `actor_object_role(object_id, role_id) WHERE is_primary`, `actor_channel(actor_id, kind_id) WHERE is_primary` | At-most-one-primary enforcement. |
| **GIN on JSONB** | `app_user_profile.preferences`, `object.extra` | JSONB containment queries. |
| **Composite** | `object_version(object_id, created_at DESC)`, `publication_object(workflow_status, publication_id)` | Common query patterns. |

|-------|----------------|-------|
| `base` | `id`, `type`, `status`, `is_editing`, `name`, `region_code`, `created_at`, `updated_at`, `updated_at_source`, `published_at`, `commercial_visibility`, `current_membership` | Root identity, workflow, and freshness contract. |

## Appendix E — Known Intentional Asymmetries

Some differences between storage shape and API shape are deliberate and should be documented rather than “corrected” in clients:

| Topic | Current behavior | Why it matters |
|-------|------------------|----------------|
| `secondary_types` | Transitional, non-canonical, opt-in column on `object`. No API automatically filters on it. | Clients must not treat it as the authoritative type model. |
| Taxonomy vs. classifications | Both rely on reference vocabularies, but the API exposes `taxonomy` separately from `classifications`. | Explorer, rendering, and indexing logic should not merge them conceptually. |
| Singular tables serialized as arrays | `object_origin` and `object_group_policy` are 1:1 in current DDL, yet payloads may serialize `origins` and `group_policies` as arrays. | Client code should tolerate future expansion to 1:N without breaking shape assumptions. |
| Cache token encoding | `cached_classification_codes` stores `scheme_code:value_code`; `cached_taxonomy_codes` stores `domain:code`. | Search, indexing, and filter builders must emit/expect composite tokens rather than raw single codes. |
