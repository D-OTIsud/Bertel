Phase 7 · Implémentation 7.5

Listes & référentiels · éditeur des vocabulaires ref_code

Le premier endroit où un super-administrateur édite directement les vocabulaires partagés de la plateforme (régimes alimentaires, allergènes, moyens de paiement, types de lit, sujets de demande…). Nouveauté la plus novatrice de la refonte : ces listes sont en base sous RLS stricte service_role/admin uniquement, donc aucune écriture directe possible depuis le navigateur. L'éditeur deux-panneaux s'appuie sur de nouveaux RPC SECURITY DEFINER gated api.is_platform_superuser(), et applique les garde-fous : code verrouillé après création, désactiver-plutôt-que-supprimer, domaines structurels en lecture seule.

Fonctionnalités & portée

Premier éditeur des vocabulaires ref_code (≈ 50 domaines indexés par ref_code_domain_registry). Master/detail : une « liste » (= un domaine) à gauche, ses valeurs éditables à droite. Le point dur est entièrement back-end : les écritures ref_* sont impossibles depuis le navigateur.

Comportement

  • Master/detail : domaines (groupés Restauration / Éditeur d'objet / CRM / Structurel) → valeurs du domaine sélectionné.
  • Par valeur : code (mono, verrouillé après création) / libellé FR + i18n (EN/DE) / position (glisser pour réordonner) / actif (bascule) / « utilisé par N fiches ».
  • Désactiver, pas supprimer, par défaut : une valeur référencée se désactive (is_active=false) · elle quitte les nouveaux choix sans casser les fiches.
  • Suppression définitive uniquement à 0 référence (bouton .btn--danger activé seulement sur une ligne à 0 usage).
  • Domaines structurels en lecture seule (facettes / taxonomies / couplage object_type) : pas de bascule, pas d'actions, pas d'« Ajouter ».
  • Ajout : le formulaire prend un libellé, slugifie un code proposé (vérifié unique), puis le code devient immuable.

Données & back-end (le point dur)

  • Écritures ref_* impossibles directement : chaque write policy = auth.role() IN ('service_role','admin') (rls_policies.sql:1649). Le client anon/authenticated ne peut jamais écrire une ligne ref.
  • ⇒ Nouveaux RPC SECURITY DEFINER gated api.is_platform_superuser() : rpc_upsert_ref_code · rpc_set_ref_code_active · rpc_reorder_ref_code · rpc_delete_ref_code (refuse si references > 0).
  • Lectures directes off ref_code_domain_registry + ref_code. « Utilisé par N fiches » = comptage RPC.
  • Précédent : api.create_tag / api.set_tag_color (même motif gated-DEFINER).
  • Respecter la normalisation du code (name_normalized générée), name_i18n, position, parent_id.

Fichiers

  • Nouveaux src/features/settings/ReferenceLists/* : DomainList, CodesTable, CodeEditModal.
  • src/services/reference-data.ts : lectures directes + appels des 4 RPC.
  • Une migration SQL : les 4 RPC + grants RLS-safe (EXECUTE pour le rôle qui appelle, P0.3) + un test CI.
  • Rendu comme panneau « Listes & référentiels » dans le hub Paramètres (7.1).

Critères d'acceptation

  • Un super-admin crée / désactive / réordonne une valeur ref_code.
  • Impossible de supprimer une valeur référencée (le RPC refuse, le bouton est désactivé).
  • Domaines structurels en lecture seule (aucun contrôle d'écriture rendu).
  • Code auto-généré puis verrouillé après création ; i18n éditable.
  • :focus-visible sur la liste, la table, les bascules et la modale.
  • Toutes les écritures via RPC DEFINER · aucune écriture PostgREST directe.
stairs Séquence : les RPC back-end + les tests CI d'abord (dépendance bloquante), puis l'éditeur deux-panneaux. v1 = domaines plats non structurels, désactiver-pas-supprimer.
